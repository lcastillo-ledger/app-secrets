# Encrypted File Format Specification

This document defines a Ledger-backed encrypted file container for files streamed through a Ledger-style application.

## Goals

1. Keep long-term encryption keys inside the Secure Element.
2. Authenticate all metadata shown to the user.
3. Use one authenticated encryption mode: AES-256-GCM.
4. Use random nonces, not deterministic nonces derived from file contents.
5. Reuse Ledger common descriptor fields and Ledger SDK cryptographic primitives where possible.

## High-level Layout

An encrypted file is a single binary blob:

| Description | Length |
| --- | --- |
| Magic word | 8 |
| Metadata TLV length | 2, big-endian |
| Metadata TLVs | variable |
| Ciphertext | variable |
| GCM tag | 16 |

`Metadata TLV length` is the byte length of the following DER-encoded metadata TLVs. The AES-GCM Additional Authenticated Data (AAD) is exactly:

```text
Magic || Metadata TLV length || Metadata TLVs
```

The ciphertext starts immediately after the metadata TLVs. The final 16 bytes of the file are the AES-GCM authentication tag.

## Magic Word

The file starts with the 8-byte ASCII string `LDG_ENCR`.

| ASCII | Hex |
| --- | --- |
| `LDG_ENCR` | `4C 44 47 5F 45 4E 43 52` |

## Metadata Authentication

There is no metadata signature in this format.

AES-GCM already authenticates the metadata because the full metadata header is used as AAD. A separate signature would only be useful if files needed public/offline metadata verification without the seed-derived encryption key. That is not a v1 goal, so the signature TLV is removed to keep the format smaller and avoid two independent integrity mechanisms.

## Byte Order

All integer values inside TLV values use big-endian encoding unless a field explicitly specifies another encoding.

## TLV Encoding

Metadata uses the Ledger common-fields TLV encoding.

| Description | Length |
| --- | --- |
| Tag | DER encoded integer |
| Length | DER encoded integer |
| Value | `Length` bytes |

Tag and length MUST be DER encoded, not fixed-size `u8`/`u16` fields. For values from `0x00` to `0x7f`, the DER encoding is a single byte. For larger values, use the long form: `0x80 | byte_count` followed by the minimal big-endian representation of the value.

TLVs are concatenated in the order chosen by the application, except `STRUCTURE_TYPE` and `VERSION` MUST be the first two TLVs.

Implementations SHOULD reuse the Ledger SDK/common TLV parser when available rather than adding a second ad hoc parser.

## Metadata TLVs

The format reuses tags from the common Nano certificates and descriptors field registry when a matching field exists. File-specific tags are allocated above the current common-field registry to avoid conflicts.

| Field | Tag | Value length | Value type | Required | Description |
| --- | --- | --- | --- | --- | --- |
| `STRUCTURE_TYPE` | `0x01` | 1 | enum | Yes | Must be `TYPE_FILE_METADATA = 0x34`, proposed as the next value after the currently assigned common `StructType` values. |
| `VERSION` | `0x02` | 1 | `u1` | Yes | Must be `1` for this version. |
| `TRUSTED_NAME` | `0x20` | variable | UTF-8 | Yes | User-visible display name. Recommended maximum: 64 bytes. |
| `TIME` | `0x26` | 4 | `s4` | Yes | Creation time, encoded as Unix time in seconds. |
| `CIPHER_SUITE` | `0x0101` | 1 | enum | Yes | Must be `1` for `AES-256-GCM-HKDF-SHA256`. |
| `MIME_TYPE` | `0x0102` | variable | UTF-8 | Yes | MIME type such as `text/plain` or `application/octet-stream`. |
| `NONCE` | `0x0103` | 12 | `bytes` | Yes | Random 96-bit AES-GCM nonce generated by the device. |

Additional TLVs MAY be added later. They are authenticated if they are included in the metadata TLV header, because the whole header is AAD.

## SLIP-10 Key Derivation

The application uses the same long-term derivation model as SMFS/Ledger Encrypt: one seed-derived root and HKDF domain separation for per-purpose keys.

| Purpose | Path |
| --- | --- |
| File encryption root | `m/44'/0xE2C1'/0'/0/0` |

Curve: Ed25519 SLIP-10 as exposed by the Ledger SDK. The derived private-key half is used as HKDF input keying material. The chain code is not used for file encryption.

This path is recoverable on any Ledger restored from the same seed. That is intentional: encrypted files are seed-recoverable, not bound to one physical device.

## Encryption Key Derivation

Input values:

| Description | Source |
| --- | --- |
| `ikm` | 32-byte private-key half derived from `m/44'/0xE2C1'/0'/0/0` |
| `salt` | ASCII `ledger-encrypt v1` |
| `info` | ASCII `ledger-encrypt v1 file aes-256-gcm` |

Derivation:

```text
aes_key = HKDF-SHA256(
    ikm = slip10_private_key_half,
    salt = "ledger-encrypt v1",
    info = "ledger-encrypt v1 file aes-256-gcm",
    length = 32
)
```

HKDF SHOULD be implemented with Ledger SDK HMAC-SHA256/SHA-256 primitives. The derived AES key MUST be cleared from memory as soon as possible.

## Content Encryption Algorithm

Version 1 supports exactly one cipher suite:

| `CIPHER_SUITE` | Algorithm |
| --- | --- |
| `1` | `AES-256-GCM-HKDF-SHA256` |

AES-GCM parameters:

| Parameter | Value |
| --- | --- |
| Key size | 32 bytes |
| Nonce size | 12 bytes |
| Tag size | 16 bytes |
| AAD | `Magic || Metadata TLV length || Metadata TLVs` |

The nonce MUST be generated randomly by the device using the Ledger SE CSPRNG for each encryption. Nonces MUST NOT be derived deterministically from file contents or metadata.

Implementations SHOULD use the Ledger SDK AES-GCM primitives, for example the `cx_aes_gcm_*` API where available.

## Encryption Checks

During encryption, the device:

1. Prompts user consent using `TRUSTED_NAME`, `MIME_TYPE`, and `TIME` when useful.
2. Generates a random 12-byte `NONCE` and adds it to metadata.
3. Derives the AES-256-GCM key from the SLIP-10 path and HKDF domain strings.
4. Starts AES-GCM with AAD equal to the full metadata header.
5. Streams plaintext, encrypts it, and returns ciphertext chunks.
6. On the final chunk, returns the GCM tag.

## Decryption Checks

During decryption, the device:

1. Parses metadata using DER TLV rules.
2. Rejects malformed metadata before prompting the user.
3. Prompts user consent using verified syntactic metadata. Cryptographic authenticity is finalized at GCM tag verification.
4. Derives the AES-256-GCM key from the SLIP-10 path and HKDF domain strings.
5. Starts AES-GCM with AAD equal to the full metadata header.
6. Streams ciphertext, decrypts it, and returns plaintext chunks as provisional plaintext.
7. On the final chunk, verifies the GCM tag. If verification fails, host software MUST discard all plaintext returned by this session.

Because streaming GCM decryption releases plaintext before the final tag is verified, host software MUST treat decrypt output as provisional until the final APDU returns `0x9000`. File decryptors SHOULD write to a temporary file and atomically rename it only after tag verification succeeds.

## Test Vectors

Implementations MUST include tests for the following canonical vectors:

| Primitive | Vector source |
| --- | --- |

| AES-256-GCM | NIST SP 800-38D AES-GCM test cases with 96-bit IV and 128-bit tag |
| SLIP-10 Ed25519 | SLIP-0010 Ed25519 master/child derivation vectors |
| DER TLV parser | Single-byte and long-form tags/lengths, including tags `0x01`, `0x20`, and `0x0101` |
